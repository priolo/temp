Problem 6

Localizzation​

The reference component appears to handle the language
For this I would use the library:
i18next

  • Allows you to dynamically load languages (I don't do my code)
  • Language change is "responsive"
  • Allows interpolation
  • There are many interesting options (for example the handling of plurals)

code

An alternative, for small projects, a Dictionary without external libraries could be used.

Validation​

In this case, external libraries hardly fit a medium / large project.
So I would do the validation with a "custom hook".

The hook keeps an error-code as a string (null == no error)
This error-code can be displayed next to the component (or as a tooltip)
the error-code is easily translated and can be collected in an array.
code

The check is done on the exchange of the value.
The value is any value that the INPUT component handles
The value is evaluated with an ARRAY of rules and then any error-code is collected
code

The rules are simple LAMBDAs that take a value and determine if this value is valid. code

It is also possible to insert a specific rule code

An important requirement is to be able to validate (or reset) all COMPONENTs externally to the VIEW.
This happens for example BEFORE sending the data to the BE.
To do this the hook registers the Listeners and therefore can use validateAll
code

VIEW-STATE decoupling​

I have repeated this to myself many times
But just for the FORM it is useful to decouple the VIEW and the STATE.
The alternative is to pass the getter / setter from the FORM to each FIELD.
This should be repeated for each branch. For example, CARDs that group more FIELDs.
So here too I used the STOREs
code

Note: FORM and FOOTER components are independent and you can put them anywhere in the rendering tree with a simple copy&paste.
